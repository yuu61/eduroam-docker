## 1. サービスの関係者

| 役割 | 説明 |
|------|------|
| **機関責任者** | サービスに参加する、または参加申請を行う機関の代表者 |
| **登録担当者** | 各サービス利用機関の利用管理者からの申請を取りまとめ、本システムに申請を行う者 |
| **利用管理者** | 各サービス利用機関に所属し、証明書を申請、使用するサーバの管理者 |
| **サービス窓口** | 国立情報学研究所が実施する、サービス参加手続き、証明書発行手続きの実務を行う組織 |
| **証明書自動発行支援システム（本システム）** | 証明書の申請の受付、審査、送付、管理を代行するシステム |
| **アクセスPIN配付者** | 各サービス参加機関に所属し、利用者に対する通知を行う者 |

## 2. 参加申請フロー

1. 機関責任者はサービス参加申請書、ドメイン申請書、確認実施手順調査票を記述し、サービス窓口に郵送します。
2. サービス窓口および認証局は受領した申請書をもとに、当該機関の審査を行います。
   - 当該機関が公的なデータベース（国税庁法人番号公表サイト）または信頼できる第三者のデータベース（信用情報機関等）に存在していること
   - 機関責任者が当該機関に存在していること
   - WHOISデータベース（JPRS等）に組織名等が登録され公開されていること
   - ドメインの使用をドメイン管理者が許諾すること（→ [3. 発行前審査](#3-発行前審査)参照）
3. 適合する機関であれば、機関情報を本システムに登録します。
4. サービス窓口は参加した機関の機関責任者に参加承認をメールで通知し、登録担当者に登録担当者認証用証明書発行の通知を行います。
5. 登録担当者はメールを受信したら、サービス窓口に電話で連絡を行います。
6. サービス窓口は登録担当者を確認し、アクセスPINを通知します。
7. 登録担当者はアクセスPINを用いて証明書をダウンロードします。

## 3. 発行前審査

### 3-1. 機関責任者の「在籍確認審査」

審査時期になると「UPKI電子証明書発行サービス 利用確認書」が特定記録郵便で届く（通常は約1年に1回）。

対応手順:

1. 署名年月日を**西暦**で記載
2. 機関責任者の氏名は「署名のみ」、「署名と押印」または「記名（スタンプ等）と押印」で対応
3. 機関責任者の氏名は、用紙の左上に記載してある氏名と一致させること
4. 必要事項を記入のうえ、利用確認書の画像データ（スキャナーまたはカメラ撮影し、PDF・JPG等で保存）を認証局の郵便物発送から**30日以内**に送付

**送付先:** upki [at] secom.co.jp

> **注意:**
> - 利用確認書に記載のランダム値は「機関責任者の在籍確認」についてのみに使用するものであり、ドメイン審査で使用するランダム値とは異なる
> - 機関責任者の変更を予定している場合は、新機関責任者の役職と氏名を連絡すること。変更対応後、新たに利用確認書が送付される
> - 利用確認書は利用機関ごとに発送されるため、複数ドメインを所有している場合は1枚に全ドメイン名が記載される

### 3-2. ドメイン審査（ドメイン使用許諾承認）

以下の3方式から選択する。

#### 方式A: メール認証

以下の5つのメールアドレスのいずれかにランダム値を送信し、認証局に返信する（**2週間以内**）。

- `admin@<ドメイン名>`
- `administrator@<ドメイン名>`
- `webmaster@<ドメイン名>`
- `hostmaster@<ドメイン名>`
- `postmaster@<ドメイン名>`

> 上記5つのメールアドレスが無い場合は作成が必要。サブドメインを含むアドレス（例: `admin@ml.zzz.ac.jp`）は不可。
> 認証局へ返信する際のメールアドレスは上記以外（個人メールアドレスやグループメール）でも可。

#### 方式B: DNSメール認証（メールアドレスをDNSへ登録）

DNSレコードにメールアドレスを登録し、そこにランダム値を送信して認証局に返信する（**2週間以内**）。認証局からのメールを受信できるメールアドレスであれば何でも可。

```
_validation-contactemail.zzz.ac.jp. 3600 IN TXT "yamamoto@zzz.ac.jp"
```

> 審査完了後、削除しても残していても問題ない。残しておけば次回の審査で利用できる。

#### 方式C: DNS認証（認証局指定のランダム値をDNSへ登録）

DNSレコードに「認証局指定のランダム値」を設定する（**2週間以内**）。

```
zzz.ac.jp. 3600 IN TXT "xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
```

> - ランダム値は認証局から送付されたものを「そのまま」登録すること
> - 登録後、削除を推奨（次回審査時に前回のレコードが残っていると分かりにくくなる）
> - `xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx` の部分は16進数のランダム値

### 3-3. WHOIS廃止対応（2025年1月15日以降）

2024年10月25日より、WHOIS以外の方式を利用した臨時のドメイン審査を実施中。

従来WHOISを使用したドメイン審査を行った利用機関は、今後の審査スケジュールがずれる場合がある。

**例1: 2024年5月31日以前に前回の審査が完了している場合**

| 日付 | イベント |
|------|------|
| 2024-05-01 | ドメイン審査および在籍確認審査完了 → 本来「2025-06-02」まで発行可だが「2025-01-13」まで発行可に変更 |
| 2024-11-20 | 臨時のドメイン審査および在籍確認審査完了 → 「2025-12-21」まで発行可 |
| 2025-10-01 | 次回審査の案内開始 |

**例2: 2024年6月1日以降に前回の審査が完了している場合**

| 日付 | イベント |
|------|------|
| 2024-06-01 | ドメイン審査および在籍確認審査完了 → 本来「2025-07-02」まで発行可だが「2025-01-13」まで発行可に変更 |
| 2024-11-20 | 臨時のドメイン審査のみ完了 → 「2025-12-21」まで発行可 |
| 2025-05-01 | 次回審査の案内開始（臨時でドメイン審査のみのため、10か月後に案内） |

## 4. サーバ証明書の発行申請

### 4-1. 証明書発行申請の流れ

1. サービスに参加した機関の利用管理者は、発行申請ファイルを作成し、各機関の登録担当者へ提出します。
2. 登録担当者は利用管理者の申請内容を審査し、本システムへ発行申請ファイルをアップロードします。
3. アップロードされたデータに問題がなければ、本システムより利用管理者へ証明書取得URLを通知します。
4. 利用管理者は証明書取得URLにアクセスし証明書の取得を行います。

### 4-2. 手続きの種別

| 手続き | 実施する場合 |
|--------|-------------|
| **新規発行**（4-3） | 新規にサーバ証明書の発行を必要とする場合。主体者DNを変更する場合 |
| **更新発行**（4-4） | 主体者DN以外を変更する場合。継続利用したい場合。失効された証明書の再発行 |
| **失効**（4-5） | 証明書が不要になった場合や秘密鍵が危殆化した場合 |

### 4-3. サーバ証明書新規発行

**【利用管理者→登録担当者】**

1. 鍵ペア及びCSRの作成を行う
2. サーバ証明書新規発行申請TSVファイルを作成し、CSRを貼り付ける
3. TSVファイルを登録担当者宛にメールで送付

**【登録担当者作業】**

4. TSVファイルを利用管理者から受け取り、本人確認を行う
5. 本システムにてTSVファイルをアップロード
   - 「サーバ証明書の[発行・更新・失効]」→「サーバ証明書 発行」を選択
   - エラー発生時の動作を選択（「全件処理を中断する」推奨）
   - TSVファイルを選択し「アップロード」
   - 審査項目をポップアップで確認後「OK」

**【利用管理者作業】**

6. サーバ証明書発行受付通知メールを受信
7. 証明書取得URLにアクセスし、サーバ証明書をダウンロード（有効期限30日）

**【登録担当者作業】**

8. ダウンロード完了通知メールを受信。一定期間経過してもダウンロード完了通知が届かない場合は利用管理者へ電話連絡

**エラーメッセージと対処方法:**

| エラーメッセージ | 原因 | 対処方法 |
|---|---|---|
| 対象のドメインは許可されていないため発行できません | 審査完了していない | サービス窓口にお問い合わせ |
| 機関の証明書発行有効期限が切れています | 発行承認有効期限を超過 | サービス窓口にお問い合わせ |
| WHOISサーバへの接続に失敗しました | Whoisサーバ接続失敗 | 時間をおいて再申請。再度失敗はサービス窓口へ |
| WHOISサーバに登録されている情報と一致しません | 組織名・登録者名が変更された | サービス窓口にお問い合わせ |
| 主体者DNのLが登録された値と一致しません | 指定されたLの値で申請されていない | ST/L値一覧を参照 |
| 主体者DNのSTが登録された値と一致しません | 指定されたSTの値で申請されていない | ST/L値一覧を参照 |

### 4-4. サーバ証明書更新発行

**【利用管理者→登録担当者】**

1. 新しい鍵ペア・CSRを作成
2. サーバ証明書更新申請TSVファイルを作成（「失効対象証明書シリアル番号」は4-6で取得）
3. TSVファイルを登録担当者宛にメールで送付

**【登録担当者作業】**

4. TSVファイルを受け取り、本人確認
5. 本システムにて「サーバ証明書 更新」でアップロード

**【利用管理者作業】**

6. 発行受付通知メールを受信、証明書をダウンロード
7. 旧サーバ証明書から新サーバ証明書への置き換え作業
8. 登録担当者に置き換え完了を通知

**【登録担当者作業】**

9. 利用管理者から置き換え完了通知を受領
10. ダウンロード完了通知メールに添付のTSVファイルを失効申請としてアップロード
11. 旧サーバ証明書失効完了通知メールを受信

> **注意:** ダウンロード完了から2週間以内に旧サーバ証明書の失効申請が行われないと、本システムより失効依頼の再通知メールが送信される。

### 4-5. サーバ証明書失効

**【利用管理者作業】**

1. サーバ証明書失効申請TSVファイルを作成（TSVツール使用）
2. 登録担当者宛にメールで送付

**【登録担当者作業】**

3. TSVファイルを受け取り、本人確認
4. 本システムにて「サーバ証明書 失効」でアップロード
5. サーバ証明書失効完了通知メールを受信（登録担当者と利用管理者の両方に送信、電子署名付き）

### 4-6. サーバ証明書情報取得

以下の場合に使用する:

- 過去に申請したサーバ証明書発行・失効・更新履歴や処理状況を確認
- 更新・失効にあたり旧サーバ証明書のシリアル番号を確認
- 過去のCSR一式、発行された証明書一式を取得
- 各サーバ証明書の利用管理者情報を確認

手順: 「サーバ証明書-証明書情報一括ダウンロード」を選択し、ダウンロード画面で「ダウンロード」を選択。

## 5. 動作環境・ブラウザ設定

### 5-1. 動作環境

| ブラウザ |
|------|
| Internet Explorer 11（Windows）以上 |
| Mozilla Firefox 68 以上（Windows、OSX以上） |

前提条件:
- JavaScriptが有効であること
- Internet Explorer: ActiveXが有効、信頼済みサイト設定済み、互換表示設定済み
- Firefox: Cookieを保存する設定が有効（登録担当者）

### 5-2. Internet Explorer の設定

JavaScriptとActiveXを有効にし、以下URLを信頼済みのサイトに追加する。

```
https://scia.secomtrust.net
```

1. **信頼済みサイトへの登録**: [ツール] → [インターネットオプション] → [セキュリティ]タブ → [信頼済みサイト] → [サイト] → `https://scia.secomtrust.net` を追加
2. **JavaScriptとActiveXの設定**: [レベルのカスタマイズ] → ActiveXコントロール → [ダイアログを表示する]、アクティブスクリプト → [有効にする]、[保護モードを有効にする]のチェックを外す
3. **互換表示の設定**: [ツール] → [互換表示設定] → `secomtrust.net` を追加

### 5-3. Firefox の設定

- JavaScriptおよびCookieを保存する機能を有効にする（登録担当者）
- サービス窓口・利用管理者は事前の設定不要
