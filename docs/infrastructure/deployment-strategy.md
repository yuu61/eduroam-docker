# eduroam展開戦略: 拠点別アセスメントと導入アプローチ

## 概要

学校法人コンピュータ総合学園の2拠点（神戸情報大学院大学・神戸電子専門学校）は、ネットワークインフラの成熟度と組織構造が大きく異なる。本文書では各拠点の現状を整理し、段階的な展開戦略を定める。

#### ネットワーク不在の矛盾

自己評価・外部評価の双方で**ネットワークインフラへの言及が一切ない**。一方で以下の施策が推進されており、すべて安定したネットワーク接続を前提としている:

- **教育DX推進**: ハイブリッド型授業（Zoom等）、オンデマンド配信、オンライン環境活用
- **基幹システム刷新**: 学生・保護者向けポータルサイト（令和5年度稼働開始）、全面移行準備中
- **AIリテラシー教育**: 全学科でAI基礎教育を必修化、独自テキスト開発

外部評価委員（地域ICT推進協議会顧問・永吉一郎氏ほか）からの提言:
- 「教育DX推進に向けて**AIを業務・教育双方に浸透させるべき**」
- 「**デジタルプラットフォーム活用**による成功事例共有の仕組みを整えてみてはどうか」
- 「Zoom等の活用によりハイブリッド授業体制は整いつつあるが、**効果的な活用方針を継続的に検証する必要がある**」
- 「他校（芸術系やIT系の大学等）や外部団体等との合同プロジェクトなどの共創の場を形成してはどうか」

→ これらの提言は**安定したネットワーク接続なしには実現不可能**だが、そのインフラ基盤に対する評価・投資計画が不在。eduroam導入提案はこのギャップを突くポイントになる。

### 公開評価と現地観測の矛盾分析

以下は、自己評価報告書・学校関係者評価報告書における学校側の公式な主張と、現地調査で観測された事実を突き合わせた結果である。すべての引用は公開文書から、すべての観測事実は現地調査・無線LANスキャン・写真記録に基づく。

#### 矛盾1: 「施設・設備は教育上の必要性に十分対応できるよう整備されている」→ 自己評価**4点（適切）**

**学校側の主張**:
- 「産業界で必要とされる技術・知識を習得させるため、実習用施設・設備についても必要に応じて検討し刷新・拡充を図っている」
- 令和6年度の整備実績: 本館の机・椅子換装、北野館の空調換装、映像・音響・照明機材の整備

**観測された事実**:
- サポート完全終了（2022年）のAllied Telesis GS924M V2が現役稼働中（写真確認済み）
- サポート終了（2025年）のYAMAHA RTX3500が机上にラックマウントなしで設置（写真確認済み）
- 天井裏配線設備があるのに窓の隙間からフラットケーブル（KB-FL6ASTP-EX等）で本番運用（写真確認済み）
- アンマネージドハブが床に直置き（写真確認済み）
- ネットワーク障害が頻繁に発生し授業が中断される事態が常態化
- **机・椅子・空調は換装するが、ネットワーク機器は2006〜2012年世代のまま放置されている**

→ 設備の「刷新・拡充」はファシリティ（什器・空調）に限定されており、ネットワークインフラは評価対象にすら入っていない。IT専門学校として4点（適切）を自己評価する根拠に疑問。

#### 矛盾2: 「教育DXを全学科において推進」「ハイブリッド型授業やオンデマンド配信に取り組んでいる」

**学校側の主張**:
- 「対面・遠隔併用によるハイブリッド型授業や、オンライン環境を生かした教育形態を積極的に取り入れ」
- 「機器類の環境整備と拡充、新たな環境等の検証を継続的に行っている」
- 全学科でAIリテラシー授業を必修科目として導入

**観測された事実**:
- ネットワークが頻繁に停止し、Zoom等のリアルタイム通信は障害時に完全に途絶
- DHCPが消失する事象が発生 → ハイブリッド授業中に学生が一斉にネットワーク接続を喪失する
- Wi-Fi 4/5世代が大半、チャネル干渉あり（ch2使用）、WLCによる自動チャネル管理なし
- 同一教室でベンダー違いの3台のAPからそれぞれ別SSIDを吹いている状態

→ 「ハイブリッド授業環境の整備」を主張しているが、その基盤であるネットワークが安定稼働していない。外部評価委員も「Zoom等の活用によりハイブリッド授業体制は整いつつあるが、効果的な活用方針を継続的に検証する必要がある」と指摘しているが、問題は活用方針ではなくインフラの信頼性。

#### 矛盾3: 「情報セキュリティに係る事項をカリキュラムに組み込み指導を徹底」→ 法令等の遵守で**4点**

**学校側の主張**:
- 「違法ダウンロード等著作権や情報セキュリティに係る事項については授業等カリキュラムに組み込み指導を徹底」
- 「個人情報保護規定を定め、教職員及び学生・学校関係者への周知とその運用を図っている」

**観測された事実**:
- **暗号化なしのオープンSSIDが4つ稼働中**（SHOKUINSHITSU, WAVEMASTER-0, R1FWIFI, guest）
- 「SHOKUINSHITSU」（職員室）がオープンネットワーク — 教職員の通信が暗号化なしで傍受可能
- PSKパスワード`12345678`を使用しているSSIDが存在
- WPAv1（脆弱な暗号化）を使用しているSSIDが存在
- WPA2-Personal + TKIP（非推奨）のSSIDが稼働中
- IT_4AとIT_4Cが同一PSKを共有

→ セキュリティ教育を「徹底」と自己評価しながら、教職員自身の通信環境が暗号化されていない。学生にセキュリティを教える教育機関として、自らの運用が教育内容と矛盾。

さらに、以下のプライバシー上の問題も確認されている:

- **DHCPにより AdGuard DNS が全端末に強制配布されている**。これにより学生・教職員の全DNSクエリ（＝インターネット利用履歴）が利用者の同意なくAdGuard社（第三者）へ送信される
- 以前はGoogle Public DNS（8.8.8.8）が同様にDHCPで配布されており、第三者DNSへの転送が常態化している
- AdGuardのフィルタリングにより、Google Analytics 4のコンソール等の正規の開発ツールがブロックされる実害が発生（手動で1.1.1.1等に変更しない限り利用不可）
- AdGuard DNSのSafe Search機能が有効な場合、**YouTubeの制限付きモードが自動的に強制される**（AdGuardの仕様上、Safe SearchとYouTube制限付きモードは分離不可能）。技術的には`restrictmoderate.youtube.com`へのDNSリライトで実装される。制限付きモードではコメントが全て非表示となり、薬物・暴力・性的コンテンツに加え、LGBTQ+や政治・人権に関する教育コンテンツも誤ブロックされることが広く報告されている（EFF 2021年調査: テストチャネルの約12%が不正確にフィルタリング）
- 利用者は全員成人（専門学校生18歳以上）であり、利用規約においても利用責任は各自が負うことが定められている。**青少年インターネット環境整備法のフィルタリング義務は18歳未満が対象**であり、本校には法的義務がない。したがって当該強制は正当な学修・研究利用を不当に制限し、必要性・比例性を欠く
- ISPのDNS利用は約款により暗黙に許容される余地があるが、AdGuardのような第三者サービスへのDNSクエリ転送は、利用者の明示的な同意なしに行うべきではない
- これは**通信の秘密**（電気通信事業法第4条の趣旨）、**個人情報の第三者提供**（個人情報保護法）、および**学校自身が定めた個人情報保護規定**に抵触しうる

→ 自己評価で「個人情報保護規定を定め、教職員及び学生・学校関係者への周知とその運用を図っている」「情報セキュリティに係る事項をカリキュラムに組み込み指導を徹底」と4点をつけながら、学校自身が利用者の同意なく全DNSクエリを第三者に転送し、かつ法的義務のないコンテンツフィルタリングを成人利用者に強制している。セキュリティ教育以前に、学校のネットワーク運用自体が個人情報保護規定に違反し、学修・研究の自由を不当に制限している可能性がある。

#### 矛盾4: 「財務基盤については問題ない」→ 財務で**4点** / 設備投資の実態

**学校側の主張**:
- 「学校創立以来の長い学園経営により培われた堅実な学校運営が行われている」
- 「的確な経営判断の下に新学科の開設や専門職大学院の設置、それに伴う施設設備の拡張・充実と教員体制の整備を行ってきた」

**観測された事実**:
- 手元現金114億円、年間19億円を資産運用に回す余力がある
- 設備関係支出は年間3,630万円 — 教育研究経費5.2億円の7%に満たない
- 2006〜2012年世代のネットワーク機器を現役運用。さらに本館のデフォルトゲートウェイからnmapフィンガープリントで**YAMAHA RT58i or RT107e**（2005〜2008年製）が検出されている（後述）
- 学内ネットワークは少なくとも4年近く構成変更なし

→ 「施設設備の拡張・充実」の実態は什器・空調・音響機材に限定。手元現金114億円がありながらネットワークインフラへの投資がゼロに近い。「堅実」と「過少投資」は異なる。

#### 矛盾5: 「デジタルプラットフォーム活用」「他校との共創」を外部評価委員が提言

**外部評価委員の提言**:
- 「デジタルプラットフォーム活用による成功事例共有の仕組みを」
- 「他校（芸術系やIT系の大学等）や外部団体等との合同プロジェクトなどの共創の場を」
- 「海外連携によるオンライン授業と現地派遣を組み合わせた教育交流モデルを」
- 「AI活用リーダーを任命し、AIリテラシー研修を体系的に」

**観測された事実**:
- 他校（特にeduroam参加校）との共創にはeduroam相互認証が前提
- 海外連携・オンライン授業は安定したネットワーク接続が前提
- eduroam非対応の教育機関は国際的な学術ネットワークから事実上孤立
- 留学生356名（23カ国）がいるにもかかわらず、eduroam未対応

→ 外部評価委員の提言をすべて実現するにはネットワークインフラの刷新が不可欠だが、学校側も評価委員側もこの前提条件に触れていない。

#### 矛盾6: 「防犯体制の整備・強化は重要課題」→ 防災で**3点**

**学校側の主張**:
- 「防犯カメラ等のさらなる拡充などハード面と運用等のソフト面の両方の改善が今後の課題」

**観測された事実**:
- 防犯カメラの映像転送・保存にはネットワーク帯域が必要だが、現状のネットワークは教室利用時に頻繁に障害が発生する状態
- 12棟の校舎に防犯カメラを展開・統合監視するには、スイッチ・PoE・VLANの整備が前提

→ 防犯カメラの拡充を課題としながら、そのインフラ基盤（PoE対応スイッチ、ネットワーク帯域、集中監視環境）が未整備。

#### まとめ: 構造的な盲点

| 学校側の認識 | 実態 |
|-------------|------|
| 「施設・設備は適切（4点）」 | 什器・空調のみ。ネットワーク機器はサポート終了品が現役。2005年製ルーター（RT107e推定、CVE-2008-0524未修正）が教室のデフォルトゲートウェイ、telnet開放 |
| 「教育DXを推進」 | 基盤となるネットワークが頻繁に停止 |
| 「個人情報保護規定を運用（4点）」 | DHCPで全端末にAdGuard DNSを強制配布、利用者の同意なくDNSクエリ（利用履歴）を第三者へ送信 |
| 「情報セキュリティ教育を徹底」 | 教職員用SSIDが暗号化なし。デフォルトGWのtelnet開放 |
| 「財務基盤は問題なし（4点）」 | 手元現金114億円で設備投資は年3,630万円 |
| 「防犯カメラ拡充が課題」 | PoE対応スイッチ・ネットワーク帯域が未整備 |
| 「ハイブリッド授業環境を整備」 | DHCP消失・5分間のネットワーク断が常態化 |

自己評価・外部評価のいずれにおいても**ネットワークインフラは評価項目として認識されていない**。教育DX・AI教育・ハイブリッド授業・国際交流・防犯体制のすべてがネットワーク接続を前提としているにもかかわらず、その基盤に対する評価・投資・計画が完全に欠落している。

ただし、**現場レベルではインフラの問題は認識されている**。学校の公式Twitterアカウントがインフラの問題について「申し訳ない、学園に要望としてあげておく」と応答した事実があり、末端の教職員・広報担当はネットワークの問題を把握している。しかし「学園に要望としてあげる」という表現が示す通り、現場には対処する権限がなく、法人本部への上申しかできない構造にある。

| レイヤー | 問題の認識 | 意思決定権限 | 自己評価への反映 |
|---------|-----------|------------|----------------|
| 現場（教職員・広報） | **あり** | なし（要望を上げるのみ） | されていない |
| 法人本部（理事会・総務） | **不明** | あり（予算・投資） | されていない |
| 自己評価報告書 | **言及なし** | — | — |

つまり問題は3層構造になっている:

1. **現場は認識しているが権限がない** — 問題を知っていても「学園に上げる」ことしかできない
2. **意思決定層に到達しているかが不明** — 要望が法人本部で検討されたのか、棚上げされたのかがわからない
3. **自己評価に反映されない** — 教育環境の評価項目にネットワークインフラが含まれていないため、公式な評価プロセスでは問題が可視化されない

これは単純な「盲点」ではなく、**現場の認識が意思決定層に届かず、公式な評価プロセスからも漏れる組織構造の問題**である。eduroam導入提案では、この構造をバイパスするために法人総務・理事会への直接的なトップダウンルートが必要となる。

## 神戸電子専門学校

### 現状

#### 無線LAN（L2: AP/SSID）

- ベンダー4社混在: YAMAHA（最多）、Buffalo、ICOM、ELECOM
- 37 SSIDが乱立。教室・学科・建物ごとに個別SSIDで接続台数を分散制御している（WLCの代替としての苦肉の策）

- Wi-Fi 4/5世代が大半、802.11k/v/r非対応

詳細は `docs/ap/existing-ap-survey.md` を参照。

#### 有線・物理インフラ（L0-L1）

- **物理配線がアドホック**: 天井裏配線用の設備（配管・ケーブルラック）が存在するにもかかわらず、窓枠の隙間から隙間用フラットケーブル（KB-FL6ASTP-EX等）で配線し本番運用している箇所あり（本館で確認）。アンマネージドハブが床に直置きされている状態も確認
- **スイッチが老朽化**: **Allied Telesis CentreCOM GS924M V2**（2010年発売、**2022年1月サポート完全終了**）が北野校舎のMDF内で稼働中であることを写真で確認済み。学内で頻繁に見かける機種。MACアドレステーブル8Kエントリ、エージングタイマー300秒（5分）。MLDv2スヌーピング対応だが有効化されているかは不明（デフォルト無効の可能性が高い）。L2MCテーブルのエントリ数は非公開だが、後継のGS900MXシリーズで512エントリのため同等以下の可能性。GS924M V2の上に**Panasonic Switch-M5ePWR**（PoE 5ポートスイッチ）が増設されている状態も確認。YAMAHAスイッチ（SWX2100/SWX2110/SWX2200世代）もMLDスヌーピング非対応
- **ルーター（北野校舎）**: **YAMAHA RTX3500**（2012年発売、2025年サポート終了）が机の上にラックマウントされずに設置されている。QNAPのNASも同じ机上に配置。ケーブル整理なし
- **ルーター（本館）**: 本館のデフォルトゲートウェイ（`10.16.0.1`）に対してnmapスキャンを実施（2025年10月14日、スクリーンショットで記録済み）。検出結果:

  | 項目 | 検出値 |
  |------|--------|
  | OS fingerprint | `Yamaha RT58i or RT107e router`（CPE: `cpe:/h:yamaha:netvolante_rt58i`） |
  | MAC | `00:A0:DE:3A:6B:55` (Yamaha) — SHOKUINSHITSUのAP（`00:a0:de:98:7f:a8`）と同一OUI |
  | Port 22 | OpenSSH 7.7 (protocol 2.0) |
  | Port 23 | open, tcpwrapped — **telnet管理ポートがネットワーク上に開放** |
  | Network Distance | 1 hop — **教室ネットワークのデフォルトゲートウェイそのもの** |

  **機種の推定**: RT58iはSSH非対応のため、SSH (OpenSSH 7.7) が検出されている点から**RT107eの可能性が高い**。ただしRT107eの最終ファームウェア（Rev.8.03.94、2013年12月）に含まれるOpenSSHは7.7（2018年リリース）より古いはずであり、フィンガープリントの精度にも注意が必要。

  **RT107eの場合のリスク**:
  - 2005年10月発売、生産完了2013年8月、修理終了2018年8月。**最終ファームウェアから12年以上更新なし**
  - **CVE-2008-0524**（CSRF、CVSS 7.5 HIGH）: 管理WebGUIのCSRF脆弱性。リモートからパスワード変更等が可能
  - telnet (port 23) が開放されており、管理トラフィックが平文で傍受可能
  - 100BASE-TX止まり（GbE非対応）、VPNはIPsec対応だがDES/3DESベース（Sweet32攻撃の対象）
  - tracerouteでこの先にさらにルーターが存在するためコアルーターではないが、教室セグメントのゲートウェイとして全トラフィックが通過する位置にある
- **光ファイバ**: シングルモードファイバが敷設されている箇所あり。100GBASE-DR等への置き換えは物理的には可能だが、既存ファイバの状態（損失・曲げ等）を計測機器で検証する必要がある
- **回線が建物内で統一されていない**: 同一建物内でフロアごとにISPが異なるケースあり。複数ISP事業者のシールが貼られた引き込み線を確認済み。回線の集約・見直しが必要

これらの問題は専門的なIT管理者が不在であることに起因する。

#### 実害: ネットワーク障害による授業停止

上記のインフラ問題は理論的なリスクではなく、**実際に授業停止という形で顕在化している**。

- ネットワークが頻繁に停止し、そのたびに授業が中断される
- ルーターのIPへのpingが通ったり通らなかったりと不安定
- DHCPが消失し、端末がIPアドレスを取得できなくなる事象が発生
- **毎回5分程度で自然回復する**ため、根本対応が先送りされ続けている。完全に壊れないことが逆にリプレースの意思決定を阻害している

老朽化したスイッチ・アドホックな配線・管理不在の複合的な結果として、ネットワークの信頼性が担保できていない。障害が短時間で回復するため「動いている」と見なされ、問題が可視化されにくい構造がある。

障害発生時の対応は教員によるLANケーブルの抜き差し（物理的なリンク再確立）であり、障害の切り分けや根本原因の特定は行われていない。

**推定される根本原因: IPv6マルチキャストによるスイッチのL2MCテーブル枯渇およびフラッディング**

参考: [大規模Wi-FiネットワークでIPv6を提供する際のマルチキャスト対策](https://note.com/sasakipochi/n/n23d5c971bef1)（JANOG56 NOCメンバーによる障害分析）

##### なぜIPv6サービスを提供していないのに問題が起きるか

現代のOS（Windows/macOS/iOS/Android）はネットワーク接続時にIPv6リンクローカルアドレス（`fe80::`）を自動生成する（RFC 4862）。リンクローカルアドレスはルーターやDHCPv6サーバの存在に一切依存せず、OSのIPv6スタックが有効であれば**ネットワークに繋がった瞬間**に以下のマルチキャストが送出される:

1. **MLD Report** → `ff02::16`（solicited-nodeマルチキャストグループへの参加通知）
2. **DAD Neighbor Solicitation** → `ff02::1:ffXX:XXXX`（アドレス重複検出）
3. **Router Solicitation** → `ff02::2`（ルーター探索。応答がなくてもリンクローカルアドレスは使用可能になる）

つまり「IPv6を提供していない」つもりでも、端末側はIPv6マルチキャストを常に送出している。

##### 問題のメカニズム

```
[複合的な障害の連鎖]

(1) MLDスヌーピング非対応（根本原因）
    2006年世代のスイッチはMLDスヌーピングに非対応:
    - Cisco Catalyst 2950/3550: MLD非対応
    - Allied Telesis AT-8000S: IGMPのみ、MLD非対応
    - YAMAHA SWX2100/SWX2110: IGMPのみ、MLD非対応（現行製品でも）
    - YAMAHA SWX2200: マルチキャスト制御自体が非対応
    → IPv6マルチキャスト（33:33:xx:xx:xx:xx宛）は無条件で全ポートにフラッディング

(2) L2MCテーブルの枯渇（MLDスヌーピング対応機でも発生しうる）
    - L2MCテーブルは一般的なスイッチで1,000〜2,000エントリ（JANOG56報告）
    - 2006年以前の世代はさらに少ない可能性
    - 端末1台あたり約3〜10エントリを消費:
      リンクローカル + 安定アドレス + 一時アドレス × 各solicited-nodeグループ
    - Windowsのデフォルト: 一時アドレスのValid期間が7日間 → 最大7個の一時アドレスが同時存在
    - 教室40台の定常的L2MCテーブル消費: 約240〜520エントリ
    - VLANで適切にセグメント分割されていなければ、複数教室・フロアが同一L2ドメインとなり
      端末数の合計でテーブルが枯渇する

(3) フラッディングの連鎖
    テーブル枯渇 or MLDスヌーピング非対応
    → マルチキャストが全ポートにフラッディング
    → スイッチCPU負荷上昇
    → STP処理遅延（ループ防止が機能不全）
    → DHCP処理遅延（Discover/Offerが到達しない）
    → ネットワーク全体の不安定化
```

**補足: MACアドレステーブル（CAM/FDB）とL2MCテーブルは別のテーブル。** MACアドレステーブルはユニキャスト転送用（8K〜16Kエントリ）、L2MCテーブルはマルチキャスト転送用（1K〜2Kエントリ）。ここでの問題は主にL2MC側の枯渇またはMLDスヌーピング非対応によるフラッディングである。

##### ベンダー別MLDスヌーピング対応状況

| ベンダー | 非対応モデル | 対応モデル |
|---------|------------|-----------|
| **Cisco** | Catalyst 2950, 3550 | Catalyst 2960（要設定）, 9000シリーズ |
| **Allied Telesis** | AT-8000S, AT-8024 | **GS924M**（MLDv2対応だがデフォルト無効の可能性、L2MCテーブル≦512）, x530, x950（デフォルト有効） |
| **YAMAHA** | SWX2100, SWX2110, SWX2200 | SWX2210以上（MLDv1/v2対応） |

YAMAHA WLXシリーズAPにはマルチキャスト→ユニキャスト変換機能もND proxy機能も非対応。ArubaやCiscoのエンタープライズAPが持つIPv6マルチキャスト最適化機能が欠如している。

##### 症状との整合性

| 観測された症状 | L2MCテーブル枯渇/フラッディングでの説明 |
|--------------|--------------------------------------|
| pingが通ったり通らなかったり | フラッディングによる帯域圧迫とCPU負荷でパケット処理が遅延 |
| DHCPが消失 | CPU負荷上昇でDHCPパケットの処理が間に合わない、またはブロードキャストストーム |
| **5分で自然回復** | **MACテーブルのエージングタイマーのデフォルトが300秒（5分）** — Cisco/Allied Telesis/YAMAHAで共通のデフォルト値。エントリのエージングアウトによりテーブルに空きが発生し一時回復 |
| ケーブル抜き差しで復旧 | リンクダウンでポートのテーブルエントリがクリアされ、フラッディングが一時的に解消 |

##### 類似事例

- **JANOG56（2025年）**: 2,000人規模の接続でL2MCテーブル枯渇。RAが「半死」状態に。NOCメンバー（ネットワーク専門家集団）ですら当日気づけなかった問題
- **GIGAスクール構想**: 文部科学省調査で全校生徒が一斉利用した際に**50.7%の学校が接続障害を経験**。推奨帯域を満たしていない学校は78.4%
- **日経クロステック報告**: 「自然復旧するネットワーク接続障害が毎日発生」— 3階建てオフィス、各階40〜50名、全て同一セグメント。MACアドレステーブルの異常が原因
- **Intel NICドライバ問題**: Intel I217-LM搭載PC（Dell OptiPlex等）でスリープ復帰時に5,000〜40,000パケット/秒のMLDトラフィックが発生する既知のバグ。教室のPCがこの機種であればさらに深刻

##### 検証方法

1. 障害発生時にスイッチの `show mac-address-table count` やL2MCテーブルの使用率を確認する（スイッチの管理権限が必要）
2. パケットキャプチャでIPv6マルチキャスト（`33:33:xx:xx:xx:xx`宛フレーム）の量を測定
3. スイッチのCPU使用率をSNMP等で監視し、障害時の相関を確認

##### 現代のスイッチ/WLCで可能な対策

現在の環境では以下の対策がいずれも実行不可能であり、**リプレースの根拠**となる:

| 対策 | 内容 | 必要な機器 |
|------|------|-----------|
| MLDスヌーピング有効化 | IPv6マルチキャストをポート単位でフィルタリング | MLDスヌーピング対応スイッチ（SWX2210以上、Cat 2960以上） |
| ND proxy | APまたは収容機器がNDPを代理応答し、マルチキャストをスイッチ外に出さない | Aruba/Cisco等のエンタープライズAP |
| マルチキャスト→ユニキャスト変換 | Wi-Fiではマルチキャストがブロードキャスト扱いで最低レートで送信されるため、ユニキャスト変換が必要 | WLC対応AP |
| RA unicast conversion | Router Advertisementをマルチキャストではなくユニキャストで応答 | 対応ルーター |
| セグメント分割 | VLAN設計でL2ドメインを教室単位に分割し、1セグメントあたりの端末数を制限 | マネージドスイッチ + 適切なVLAN設計 |
| Storm Control | 異常なマルチキャストバーストを検出・抑制 | マネージドスイッチ |

#### 組織の問題（L8: 政治層）

- 学科・建物・フロアごとに管理者が異なり、横断的な意思決定が困難
- IT学科のトップが合意形成における阻害要因となるリスクがある
  - 議論のすり替え、前提条件の蒸し返し等により、技術的に正当な提案でも意思決定が停滞する可能性
- eduroam導入は「誰の管轄か」「誰が設定するか」が曖昧

### eduroam導入に必要な作業

eduroam導入だけであれば既存WPA2-Enterprise対応APにSSIDを追加する方法もあるが、インフラ全体の老朽化を考えると**ほぼ総入れ替えのリプレースが必要**。

#### 最低限（eduroam SSIDの追加のみ）

1. 既存RADIUSサーバの所在と管理者を特定
2. WPA2-Enterprise対応APにeduroam SSIDを追加（マルチSSID機能を利用）
3. 各APにRADIUSサーバのIPとshared secretを設定
4. ベンダーごとに設定方法が異なるため、YAMAHA/Buffalo/ICOM/ELECOMそれぞれの手順書が必要

#### 本来必要な範囲（インフラ刷新）

1. AP統一 + WLC導入（SSIDを3程度に集約、マルチキャスト→ユニキャスト変換・ND proxy対応）
2. L2/L3スイッチのリプレース（2006年世代の撤去、MLDスヌーピング対応・IPv6ネイティブ対応）
3. VLAN設計の見直し（教室・フロア単位でのL2セグメント分割）
4. 物理配線の正規化（隙間ケーブル→天井裏配線への移行）
5. ISP回線の集約（フロアごとにバラバラなISPを統一）
6. 既存光ファイバの品質検証と必要に応じた高速化

### eduroam対象規模

- 在学生1,755名 + 教職員218名 ≒ **約2,000人**
- 実際の端末数はPC + スマートフォンで**4,000台以上**と推定（授業用PC貸与 + 個人端末BYOD）
- 留学生356名（23カ国・地域、全体の約20%）→ **eduroamの国際相互利用メリットが特に大きい**。留学生が母国の大学でeduroamを利用していた場合、到着直後からネットワーク接続可能
- 12棟の校舎に展開する必要があり、WLC統合管理なしでの個別設定は非現実的

### 見通し

KIC（大学院）側は認証基盤の整備と設定変更のみで導入可能。一方、専門学校側はeduroam以前にネットワークインフラ全体が老朽化・断片化しており、eduroamをきっかけとしたインフラ刷新の提案が現実的。組織的な合意形成が最大の障壁であり、トップダウンルートによるバイパスが不可欠。

**提案の論拠となる公開情報**:

1. **財務**: 年間黒字12.6億円、手元現金114億円、年間19億円を資産運用に回す余力がありながら設備投資は年間3,630万円。「投資余力がない」のではなく「投資の意思決定がされていない」
2. **自己評価**: 教育環境が全11評価項目中**最低スコア（3.33/4.00）**。学校自身が施設・設備の課題を公式に認識している
3. **矛盾**: 教育DX・AIリテラシー教育・ハイブリッド授業を推進しながら、その基盤となるネットワークインフラへの言及・投資が不在
4. **外部評価**: ICT推進協議会顧問がDX・デジタルプラットフォーム活用を提言 — いずれも安定したネットワーク接続が前提
5. **募集危機**: 定員充足率がR4=116%→R7=89%と急落。留学生比率20%（23カ国）のeduroam国際相互利用は差別化要因になりうる

## 展開戦略

### 基本方針: 2拠点同時並行

```
KIC・神戸電子を同時に進める。困ったら教授と相談。
問題が起きた場合のみ KIC → 神戸電子 のカナリアリリースに切り替える。
```

#### KIC側

- 教授と連携し、Aruba WLCにeduroam SSIDを追加
- 本プロジェクトで構築するRADIUSサーバを接続
- Aruba統一環境のため技術的障壁は低い

#### 神戸電子側

- 教授経由で法人総務に掛け合い、トップダウンで導入承認を得る
  - 運営法人は同一（学校法人コンピュータ総合学園）。理事長・常務理事・校長がいずれも福岡姓の同族経営であり、トップダウンの意思決定構造
  - 総務経由で既存RADIUSの管理者も判明する可能性が高い
- 既存WPA2-Enterprise運用AP（YAMAHA/Buffalo）にeduroam SSIDを追加
- ベンダーごとの設定手順書を用意し、順次展開
- 長期的にはWLC導入によるSSID統合・AP統一管理への移行を目指す

### フォールバック: カナリアリリース

同時並行で問題が発生した場合（L8の阻害、技術的問題等）:

1. **KICで先に稼働させる**（Aruba環境で確実に動く状態を作る）
2. **KICの実績をテコに神戸電子側の障壁を突破する**（「もう動いています」は強い）
3. 神戸電子側はIT学科管轄外のAP（JCOM系学科等）から段階的に広げるルートも検討

## L8対策の原則

- **「やるかどうか」の議論に巻き込まれない**: 法人総務からの決定事項としてフレーミングする
- **技術判断の余地を最小化**: 「APにSSIDを1つ追加するだけ」レベルに矮小化して説明する
- **困ったら教授に相談**: KIC側のキーパーソン（Discord 1hop）は技術・組織両面で頼れる
- **議論の記録を残す**: ムービングゴールポストへの防御として、合意事項は文書化する
